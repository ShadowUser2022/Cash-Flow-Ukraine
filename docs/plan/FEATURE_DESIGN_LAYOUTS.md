# Feature branch: Design layouts (3 варіанти)

Ціль: підготувати **3 різні UI-лейаути** для Cash Flow Ukraine (v0), щоб потім легко вибрати/перемикати та адаптувати під різні пристрої.

## 0) Назва гілки

`design/layouts-v1`

## 1) Загальні правила (не ламати core)

- **Не змінювати** game rules / backend API / контракти.
- Зміни тільки у `frontend/` (верстка/стилі/компонування).
- Мінімізувати “перемальовку” логіки: переносимо блоки, а не переписуємо механіку.
- Усі 3 лейаути мають бути **адаптивні** (desktop/tablet/mobile).
- Має збиратись: `npm run build` в корені.

## 2) Що в UI є “блоками”

Орієнтуємось на існуючі блоки (App):

- `top-bar` (бренд + статус з’єднання)
- `status-rail` (ім’я/трек/позиція/хід)
- `event-panel` (guide + кнопка дії + pending контент)
- `finance-board` (мета + доходи/витрати)
- `board` (вікно дошки з клітинками)
- `debug modal` (не чіпати логіку, лише позицію/кнопку)

## 3) Вимога: 3 варіанти лейаута

Реалізувати 3 варіанти **як CSS-режими** (клас на `<main className="app-shell ...">`), щоб можна було легко перемикати:

- `layout-a` (default)
- `layout-b`
- `layout-c`

Перемикач для розробки (v0.1, без складної UI):

- додати query param `?layout=a|b|c` (fallback `a`)
- або `localStorage("cashflow-layout")`

## 4) Layout A — “Classic” (поточний, але відшліфувати)

Ціль: лишити знайомий вигляд, мінімальні перестановки.

- Desktop:
  - `event-panel` зверху
  - далі `finance-board`
  - далі `board`
- Mobile:
  - `event-panel`
  - `board` (3 клітинки)
  - `finance-board`

Покращення:
- вирівняти `top-bar` в один рядок на mobile (вже є)
- не допускати горизонтального скролу

## 5) Layout B — “Board-first”

Ціль: на мобільному і small tablet візуально пріоритетна дошка + подія під нею.

- Desktop:
  - 2 колонки: зліва `board`, справа `event-panel`
  - `finance-board` під ними на всю ширину
- Mobile:
  - `board` зверху (3 клітинки)
  - `event-panel` нижче
  - `finance-board` в кінці

Деталь:
- `event-panel` кнопка дії завжди видима, без “стрибу”

## 6) Layout C — “Finance-first / навчальний”

Ціль: навчальний режим: спочатку цифри → потім дія.

- Desktop:
  - зверху `finance-board` (мета + доходи/витрати)
  - далі `event-panel`
  - справа (або знизу) `board`
- Mobile:
  - `finance-board` зверху (але компактно, без зайвих підказок)
  - `event-panel`
  - `board` внизу

Деталь:
- “Баланс / міс.” завжди в зоні видимості без скролу (або sticky header всередині finance-board — якщо просто)

## 7) Acceptance criteria (чітко)

- Є 3 режими `layout-a|b|c` і вони реально змінюють компонування.
- На mobile немає горизонтального скролу; кнопка основної дії не з’їжджає.
- `board` показує 3 клітинки на mobile (як зараз), 5 на desktop.
- `npm run build` проходить.

## 8) Test plan (мінімум)

Перевірити вручну:

- Desktop ширина ~1200: всі 3 лейаути, pending deal/expense/salary.
- Mobile ширина ~390: всі 3 лейаути, і щоб `event-panel` був читабельний.
- Перемикання `?layout=b` не ламає стан гри.
