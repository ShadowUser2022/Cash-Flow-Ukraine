## 🔧 Під час розробки

### Перед кожною зміною:

- [ ] Перевірити відповідність `DEVELOPMENT_STRATEGY.md`
- [ ] Переконатися, що не порушуються базові правила Cashflow
- [ ] Перевірити TypeScript типи

### При роботі з фінансовою логікою:

- [ ] Використовувати стандартні функції: `calculatePassiveIncome()`, `checkFinancialFreedom()`
- [ ] Перевіряти формулу: `Passive Income ≥ Expenses = Financial Freedom`
- [ ] Тестувати розрахунки з різними професіями

### При роботі з ігровою механікою:

- [ ] Дотримуватися послідовності ходу з документу
- [ ] Перевіряти кількість клітинок: Rat Race = 24, Fast Track = 16
- [ ] Тестувати перехід між колами

### Базові правила Cashflow:

- ❌ НЕ змінювати умову фінансової свободи
- ❌ НЕ додавати третє коло
- ❌ НЕ міняти кількість клітинок на полі
- ❌ НЕ змінювати основну механіку двох кіл

### Архітектура:

- ❌ НЕ дублювати фінансову логіку
- ❌ НЕ створювати файли понад 200 рядків

- ❌ НЕ робити глибоку вкладеність компонентів

## 🆘 Якщо щось не так

### Проблеми з серверами:

1. `🛑 Stop All Servers`
2. `🚀 Start Dev Servers`
3. `🧪 Test Application`

### Проблеми з API:

1. Перевірити логи backend сервера
2. Перевірити, чи правильно експортуються маршрути
3. Звернутися до `DEVELOPMENT_STRATEGY.md`

### Питання щодо механіки:

1. Дивитися `GAME_RULES_AND_MECHANICS.md`
2. Перевіряти приклади реальної гри в документі
3. Консультуватися з командою
